# 需求冻结：成就日检与钓鱼后台减负

## 目标

重构 `astrbot_plugin_fishing` 的成就检查、自动钓鱼和日志清理路径，把原本高频的后台检查改为按每日刷新点集中执行，降低 `SQLite + 轮询线程` 负载，同时不改变玩家可见规则。

## 范围

- 成就自检从 `10` 分钟轮询改为按 `game.daily_reset_hour` 每日统一检查一次。
- 自动钓鱼减少重复查询和循环内重复维护操作。
- 日志过期清理从“每次写日志顺手清理”改为“每日刷新时统一清理”。
- 为自动钓鱼筛选和日志清理补充必要索引。
- 保留现有成就定义、奖励、自动钓鱼命令入口和日志保留口径。

## 冻结规则

### 1. 成就检查

- 成就系统继续保留，不删除。
- 后台成就检查线程不再每 `10` 分钟全量扫描用户。
- 成就检查改为按全局 `game.daily_reset_hour` 对齐的每日刷新周期统一执行一次。
- 插件启动后，如果当前刷新周期尚未执行过本轮成就检查，应立即补做一次。
- 成就奖励和解锁规则保持不变。

### 2. 自动钓鱼

- 自动钓鱼主循环继续保留现有冷却、成本、区域通行证与装备损坏逻辑。
- 自动钓鱼循环内不再先查“自动钓鱼用户 ID”再逐个二次查用户详情，而是直接批量取出已开启自动钓鱼的用户。
- 自动钓鱼循环内不再对同一轮用户重复做明显可合并的每日维护检查。
- 自动钓鱼实际执行钓鱼时，应复用循环内已取出的用户对象，避免同一轮内重复读取同一用户。

### 3. 日志清理

- 不再在每次写入钓鱼、抽卡、擦弹、签到日志时执行全局过期清理。
- 这些日志的“保留每用户最近 `50` 条”和“清理 `30` 天前数据”规则继续保留。
- 日志清理改为在每日刷新周期统一执行一次。
- 税收记录等更偏账本性质的数据不纳入本次集中清理范围。

### 4. 数据库优化

- 为 `users.auto_fishing_enabled` 增加索引。
- 为高频按时间清理的日志表增加时间字段索引。
- 不改用外部数据库。
- 不引入 Redis、任务队列或额外后台进程。

## 验收标准

- 成就线程不再按 `10` 分钟频率全量扫描用户。
- 在一个刷新周期内，成就全量检查最多执行一次；跨过下一刷新点后可再次执行。
- 自动钓鱼循环可直接获取自动钓鱼用户对象，不再对每个自动钓鱼用户额外执行一次 `get_by_id`。
- 自动钓鱼循环中的每日维护逻辑不会在同一轮内对每个自动钓鱼用户重复触发。
- 写入钓鱼、抽卡、擦弹、签到日志时，不再附带全局 `30` 天清理。
- 每日刷新维护执行时，会统一完成日志过期清理与每用户最近 `50` 条截断。
- 数据库迁移后，`auto_fishing_enabled` 查询和日志按时间清理具备对应索引。

## 非目标

- 不重做成就定义和成就奖励内容。
- 不把成就改成完全事件驱动。
- 不重做区域通行证、税收、交易所或 WebUI。
- 不更改日志保留天数和每用户保留条数。
