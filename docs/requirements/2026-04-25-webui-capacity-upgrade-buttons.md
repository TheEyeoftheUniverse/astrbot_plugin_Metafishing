# WebUI 容量升级按钮需求

- 日期：2026-04-25
- 范围：`player` WebUI 的鱼塘页与水族箱页
- 目标：为鱼塘和水族箱的容量卡片增加与期货页面一致的升级入口，展示下一级容量与升级花费，并可直接在页面内发起升级。

## 需求冻结

### Goal
让玩家在鱼塘页和水族箱页直接看到容量成长信息，并像期货页一样一键升级。

### Deliverable
- 鱼塘页面新增容量卡片，显示当前占用、当前容量、下一级容量、花费和升级按钮。
- 水族箱页面现有容量卡片补充下一级容量、花费和升级按钮。
- WebUI 新增对应升级 API，按钮点击后调用后端升级并刷新页面。

### Constraints
- 复用已有升级逻辑，不重写鱼塘或水族箱升级规则。
- 前端风格应与期货页面升级按钮保持一致。
- 不改动现有 README 的用户改动。

### Acceptance Criteria
- 鱼塘页可见容量卡片，且在存在下一档升级时显示“下一级 + 花费 + 升级按钮”。
- 水族箱页容量卡片显示同样的升级信息。
- 点击按钮能调用对应 API，成功后提示并刷新，失败时展示失败原因。
- 达到最高等级时不显示升级按钮，改为显示已满级文案。

### Product Acceptance Criteria
- 玩家无需跳转或记忆指令即可理解当前容量成长路径。
- 文案与期货页保持一致的理解成本。

### Manual Spot Checks
- 登录后打开 `/fishpond` 检查容量卡片文案和按钮。
- 登录后打开 `/aquarium` 检查容量卡片文案和按钮。
- 手动核对按钮调用的 API 路径与成功后的 reload 逻辑。

### Completion Language Policy
- 仅在代码改动完成且完成基本静态验证后，才声明任务完成。

### Delivery Truth Contract
- 以仓库内代码 diff、需求文档、计划文档和运行收据作为完成依据。

### Non-goals
- 不调整鱼塘/水族箱的升级数值。
- 不重做页面视觉体系。
- 不扩展到首页升级面板或管理端页面。

### Autonomy Mode
- 直接实现，除非发现现有页面结构与需求冲突。

### Inferred Assumptions
- “像期货的升级按钮一样”指信息结构和交互一致，不要求完全相同的配色。
- 水族箱升级费用以金币为主，但若未来存在钻石费用，页面应一并展示。
