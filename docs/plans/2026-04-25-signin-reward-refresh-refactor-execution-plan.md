# 执行计划：签到奖励与刷新口径重构

## 内部执行等级

M：签到逻辑集中在 `UserService`，周边只需要同步“今日已签到”状态查询口径。

## 步骤

1. 冻结本次需求文档，确认奖励、连签、不封顶和刷新口径规则。
2. 新增统一刷新周期标记：
   - 基于 `daily_reset_hour` 计算当前刷新周期。
   - 复用现有 `check_ins.check_in_date` 存储刷新周期标记日期。
3. 重写签到奖励：
   - 移除连签节点额外金币和每日免费池抽取。
   - 签到成功后先增加连签天数，再按新天数发放 `random(100, 300) * 天数` 金币和 `1 * 天数` 高级货币。
4. 同步入口和展示状态：
   - QQ 命令失败时也返回服务消息。
   - 玩家 WebUI、用户端 API、状态图使用相同刷新周期判断是否已签到。
5. 做静态和行为验证。

## 验证命令

- `python3 -m py_compile core/utils.py core/services/user_service.py handlers/common_handlers.py manager/user_api.py manager/user_server.py player/server.py draw/state.py`
- 轻量行为脚本验证：
  - 第 1 天奖励为 `100-300` 金币和 `1` 高级货币。
  - 连续第 30 天当天奖励为 `3000-9000` 金币和 `30` 高级货币。
  - 30 天累计高级货币为 `465`，金币在随机区间内。
  - 同一刷新周期重复签到被拒绝。
  - 签到过程不调用免费抽卡服务。
